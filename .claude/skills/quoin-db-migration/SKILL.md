# Modifying the QuoinAPI Schema

A schema change here is a five-move loop: edit the model → autogenerate → **review** → apply → write tests. The review step is the one people skip and regret. Long-form reference is in [docs/guides/database-migrations.md](../../../docs/guides/database-migrations.md). The PreToolUse hook (`.claude/hooks/block-sensitive.sh`) will refuse hand-edits to existing files in `alembic/versions/` — that's intentional, not a bug.

## Workflow

### 1. Edit the SQLModel

Make the change in the appropriate `app/modules/<module>/models.py`. Keep the change atomic — one logical schema change per migration. If the user asks for two unrelated changes (e.g. "add an index AND drop a column"), generate two separate migrations.

### 2. Generate the migration

```bash
just migrate-gen "describe the change in present tense"
```

Good messages: `"add email_verified to user"`, `"index user.created_at"`, `"drop legacy session_token from user"`. Bad: `"changes"`, `"fix"`, `"update"`. The message becomes the docstring of the migration file.

### 3. Review the generated script — this is the important step

Open the new file in `alembic/versions/` and read it line by line. Autogenerate is good but not infallible. Check:

- **Did it pick up exactly the columns/indexes/constraints you intended?** Diff your model change against the migration; anything that doesn't correspond is suspicious.
- **Did it accidentally pick up unrelated drift?** If `op.drop_table(...)` or `op.drop_column(...)` appears for something you didn't touch, your local DB is out of sync with `head` — abort and resync, don't apply.
- **NOT NULL columns on existing tables need a default or a backfill.** Adding `nullable=False` to a populated table without a `server_default` (or a two-step migration) will fail or lock the table on production data. Either set a `server_default`, or split into: add nullable → backfill → make non-null.
- **Type narrowing loses data.** `String(255) → String(50)` truncates silently; `Integer → SmallInteger` overflows. Reject these unless the user has explicitly accepted the data loss.
- **Enum changes are not autogenerated reliably.** Postgres ENUM types need explicit `ALTER TYPE ... ADD VALUE` ops; autogen often misses them. Hand-write the up/down for enum modifications.
- **The `downgrade()` function is reversible.** If `downgrade()` is empty or a `pass`, decide whether that's acceptable (drop migrations are sometimes intentionally one-way). If the operation should be reversible, write the inverse.
- **Server defaults vs. Python defaults.** `Field(default=...)` is Python-side only — the DB sees nothing. Use `sa_column_kwargs={"server_default": "..."}` when the database needs to fill the value (e.g. for backfills).

### 4. Apply the migration

```bash
just migrate-up
```

If it fails partway through, **do not edit the migration file to "fix" it** — the hook will block you anyway. Instead: `just migrate-down`, fix the model, regenerate.

### 5. Write tests for the new schema state

The model change is real now. If new fields, types, or constraints affect business logic:
- Add or update repository tests covering the new column.
- If a new constraint can be violated (unique, not-null, check), test that the right domain exception is raised.

Run `just check` to confirm nothing broke.

### 6. Commit

The migration file and the model change go in the **same commit**. If they land separately, anyone who pulls the model without the migration (or vice versa) will see a broken app.

## When the autogen output looks wrong

Don't fight it — delete the migration file (`rm alembic/versions/<the_new_one>.py`) before it's committed, fix the model, and regenerate. The PreToolUse hook only blocks Edit/Write; you can still `rm` an unwanted file.

If the migration is already committed but never released, write a *new* migration that corrects course — don't go back and rewrite history.

## Things that bite

- **Autogen + hand-edits in alembic/env.py drift.** If `env.py` filters tables (e.g. `include_object`), autogen won't see what you expect. Check `alembic/env.py` once if results are surprising.
- **Forgetting to import the model.** If a new module's model isn't imported anywhere that `Base.metadata` sees, autogen produces an empty migration. The fix: ensure the module is registered (the `quoin-new-module` skill covers this for fresh modules).
- **Running migrate-up against the wrong DB.** `just db` brings up the dev Postgres on the port in `.env`. If you have multiple Postgres instances, double-check you're hitting the right one before applying.
